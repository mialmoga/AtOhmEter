Construye tu universo propio

AtOhmEter es un laboratorio de física especulativa en el navegador. Cualquier campo, cualquier regla, cualquier idea — si implementa el contrato del shell, funciona. Este manual te enseña cómo.

4 archivos por engine

Sin dependencias
✦ Por dónde empezar — para humanos
1. Escribe el paper primero. ¿Qué hace tu campo? ¿Cuál es su ecuación? ¿Qué estados tiene? Rellena paper.json y config.json antes de tocar el motor.

2. Implementa lo que prometiste. El motor.js tiene que hacer lo que el paper dice. Si no coinciden, uno de los dos está mal — y esa tensión es donde se aprende.

3. Copia un shader existente. No empieces por el GLSL. Copia engines/mirror/shader.glsl y ya.

4. Si carga sin errores: tu IA (o tú) está certificada para generar universos de bolsillo. 🦝

01 Filosofía

El laboratorio no juzga qué física es "real". Los motores existentes incluyen campos que mueren por entropía irreversible, autómatas celulares de consenso social, campos con memoria temporal estratificada, y partículas que heredan fase al reproducirse.

La única restricción es técnica: implementar el contrato de la API para que el shell sepa cómo llamar tu motor, visualizarlo, y guardar snapshots.

La pregunta que importa
¿Qué pasa si el espacio tiene esta propiedad? ¿Y si los campos se comportan así? El simulador te dice qué pasa. Tú decides qué preguntar.

02 Estructura de archivos

Cada engine vive en su propia carpeta dentro de engines/:

engines/
└── mi_engine/
    ├── motor.js      ← tu física (el único archivo que realmente importa)
    ├── shader.glsl   ← el render (puedes reusar uno existente)
    ├── config.json   ← parámetros, semillas, HUD
    └── paper.json    ← narrativa, fórmulas, tooltips

Se carga con: ?engine=mi_engine

Shader reutilizable
Si no quieres escribir GLSL, copia engines/mirror/shader.glsl. Usa rueda completa de colores por fase — funciona bien para la mayoría de campos. Para colores semánticos (como COSTUMBRE) necesitas tu propio shader.

03 API obligatoria

El shell importa createEngine y espera exactamente estas funciones en el objeto devuelto. Nombres exactos, sin variaciones. 

export function createEngine(N, renderVolume, phaseData, texture3D, texturePhase) {
  // N             — tamaño del grid (32, 48 o 64)
  // renderVolume  — Float32Array(N³) — escribe aquí el brillo del vóxel
  // phaseData     — Float32Array(N³) — escribe aquí el color/fase
  // texture3D     — THREE.Data3DTexture — marca needsUpdate=true
  // texturePhase  — THREE.Data3DTexture — ídem

  return {
    step(),           // avanzar un frame — llamado cada frame
    refresh(),        // escribir al GPU — llamado según refreshRate
    getMetrics(),     // devolver objeto de métricas
    seed(name),       // inicializar semilla por nombre
    inject(name),     // ejecutar inyección
    applyParams(p),   // recibir cambios de sliders
    getParams(),      // devolver objeto de parámetros
    getState(),       // snapshot serializable
    setState(s),      // restaurar snapshot
    savePrev(),       // puede estar vacío: () => {}
    classifyState(m), // string de estado desde métricas
  };
}
step()
Avanza un paso de física. El shell lo llama cada frame de animación. Toda tu ecuación va aquí.
refresh()
Escribe en renderVolume y phaseData, y marca texture3D.needsUpdate = true. Sin esto el render no se actualiza.
seed(name)
Inicializa el campo con la semilla indicada. Llamado en Reset y al cambiar semilla. Debe llamar refresh() al final.
savePrev()
Puede estar completamente vacío: savePrev() {}. El shell lo llama pero no requiere nada.

05 classifyState(m)

Recibe el objeto de métricas y devuelve un string. El shell lo usa para actualizar la narrativa del Live Paper y las opacidades de la fórmula.

Los 7 estados con estilos predefinidos en el shell:

function classifyState(m) {
  if (m.psiMax > 10)        return 'collapse';   // divergencia
  if (m.boundary > 0.5)     return 'locked';    // estructura fija
  if (m.coherence > 0.7)    return 'stable';    // identidad preservada
  if (m.E_total > 1.0)      return 'pumping';   // alta energía
  if (m.E_total > 0.3)      return 'active';    // campo activo
  if (m.E_total > 0.05)     return 'nucleating';// emergiendo
  return 'vacuum';                               // reposo
}

Puedes agregar estados custom en paper.json — solo asegúrate de definirlos en states, stateLabels y formulaOpacity.

08b Capacidades del shell

El shell ofrece cuatro entradas opcionales que conectan tu motor con el mundo físico del teléfono. Todas son opt-in: se activan con una bandera en config.json, y si no las declaras no pasa nada (cero impacto en tu motor). Cada una muestra su propia barra.

🎤 Micrófono — el campo escucha

Bloque mic en config. Mapea las features del audio (volume, bass, mid, treble, todas 0..1) a un param (lo modula entre min..max) o a una inject (la dispara al cruzar threshold). 

"mic": {
  "enabled": true,
  "bindings": [
    { "feature": "bass",   "param": "MI_PARAM", "min": 0, "max": 2 },
    { "feature": "volume", "inject": "soplar", "threshold": 0.12,
      "cooldown": 3, "passStrength": true }
  ]
}

passStrength: true pasa la intensidad como data.strength a tu inject(name, data) — soplar más fuerte, efecto más grande. BURBUJA usa esto: tu aliento crea e infla las pompas.

📱 Giroscopio — el campo siente

Bloque gyro. Mapea la orientación del teléfono (features -1..1) a params. tiltX (izq/der) y tiltY (ad/atrás) son confiables; tiltZ (brújula) es inestable — úsalo solo para efectos no críticos. 

"gyro": {
  "enabled": true,
  "bindings": [
    { "feature": "tiltX", "param": "GRAV_X", "min": -0.06, "max": 0.06 },
    { "feature": "tiltY", "param": "GRAV_Y", "min": -0.06, "max": 0.06 }
  ]
}
⚠ El eje Y va al revés
En pantalla, +Y es arriba. Si quieres que "inclinar hacia ti" haga caer las cosas, invierte el signo en tu motor: a.vy += -gy. (Lo aprendimos con BLOOM: caían hacia arriba. 🙃)

🔄 Órbita automática — el visor contempla

"autoOrbit": true y "autoOrbitSpeed": 0.5. La cámara gira sola para apreciar el volumen desde todos los ángulos. Es un control universal (toggle + slider debajo del timeline) que funciona en todos los motores; la bandera solo fija el estado inicial. Sigue girando con la simulación pausada — combínalo con el replay de snapshots para una pieza de museo: la física avanza y la cámara la rodea.

👆 Toque — el dedo interactúa

El shell ya trae raycaster. Si tu motor implementa el caso 'touch' en inject, recibe el punto tocado en coordenadas locales [-1,1]³: 

function inject(name, data) {
  if (name === 'touch') {
    const gx = (data.x*0.5+0.5)*N; // [-1,1] → [0,N]
    const gy = (data.y*0.5+0.5)*N;
    const gz = (data.z*0.5+0.5)*N;
    // usa el punto: atractor, tinta, calor...
  }
}

Se llama repetido al arrastrar; no avisa al soltar. Para saber cuándo termina, usa un valor que decaiga solo y se renueve con cada toque. BLOOM e INK usan este mecanismo.

📷 Dirección de cámara

El shell expone window._camera. Útil si tu motor reacciona a desde dónde lo miran (BURBUJA detecta qué cara del cubo está al frente). Protégelo con if (cam) para correr headless.

Plantilla lista
motor_v2.js ejemplifica las cuatro: recibe el toque, lee la cámara, y tiene params que el micro y el giro modulan. Cópialo si tu motor usa estas entradas.

09 Patrones comunes

Índice 3D periódico

function idx(x, y, z) {
  return ((x+N)%N)*N*N + ((y+N)%N)*N + ((z+N)%N);
}

Laplaciano estándar

function laplaciano(F, out) {
  for (let x=0;x<N;x++) for (let y=0;y<N;y++) for (let z=0;z<N;z++) {
    const i = idx(x,y,z);
    out[i] = F[idx(x+1,y,z)] + F[idx(x-1,y,z)]
           + F[idx(x,y+1,z)] + F[idx(x,y-1,z)]
           + F[idx(x,y,z+1)] + F[idx(x,y,z-1)]
           - 6*F[i];
  }
}

Gradiente

function gradiente(F, gx, gy, gz) {
  for (let x=0;x<N;x++) for (let y=0;y<N;y++) for (let z=0;z<N;z++) {
    const i = idx(x,y,z);
    gx[i] = (F[idx(x+1,y,z)] - F[idx(x-1,y,z)]) * 0.5;
    gy[i] = (F[idx(x,y+1,z)] - F[idx(x,y-1,z)]) * 0.5;
    gz[i] = (F[idx(x,y,z+1)] - F[idx(x,y,z-1)]) * 0.5;
  }
}

Pulso gaussiano (semilla típica)

const c = N >> 1; // centro del grid
for (let x=0;x<N;x++) for (let y=0;y<N;y++) for (let z=0;z<N;z++) {
  const i = idx(x,y,z);
  const dx=x-c, dy=y-c, dz=z-c;
  field[i] = Math.exp(-(dx*dx+dy*dy+dz*dz) / (2*(N/5)**2));
}

13 Motor V1 — El Campo Resonante

Un motor completo con personalidad: campo oscilante que se amplifica donde sus vecinos vibran en fase y se inhibe donde vibran en antifase. Las estructuras emergen donde la resonancia se auto-sostiene.

Ecuación: ∂²ψ/∂t² = c²∇²ψ − ω²ψ + κ·C(ψ)·ψ
donde C(ψ) es la correlación local de fase.

Cómo usar
Copia motor_v1.js a engines/resonante/. Crea los config.json y paper.json correspondientes (guíate por la sección 06 y 07). Copia cualquier shader.glsl existente. Carga con ?engine=resonante.

El archivo motor_v1.js incluye:

  • Campo complejo ψ con parte real e imaginaria
  • Segundo orden temporal (velocidades + Verlet)
  • Observable emergente: coherencia local de fase
  • 3 semillas: pulso gaussiano, vórtice, ruido
  • 3 inyecciones: sacudir, antifase, pulso puntual
  • Overlap de modo como medida de identidad
  • Todos los aliases necesarios (seed, setState, loadState)
